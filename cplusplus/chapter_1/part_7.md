### INTERVIEW_QA: Theoretical and Design Questions

#### Q1: What is the Rule of 5 in C++? Why was it introduced?
**Difficulty:** #intermediate  
**Category:** #design_pattern #memory #interview_favorite  
**Concepts:** #rule_of_five #move_semantics #resource_management #constructors #destructors

**Answer:**
The Rule of 5 states that if a class defines any of destructor, copy constructor, copy assignment, move constructor, or move assignment, it should define all five.

**Explanation:**
Introduced with C++11 to include move semantics, the Rule of 5 extends the Rule of 3. Classes managing resources need:
1. Destructor
2. Copy constructor
3. Copy assignment operator
4. Move constructor
5. Move assignment operator

**Follow-up:** The Rule of 3 (C++98/03) only required the first three. Move semantics were added to enable efficient resource transfer from temporary objects.

**Key takeaway:** Resource-managing classes need all five special members for both safe copying and efficient moving.

#### Q2: If you declare a move constructor, what happens to the copy constructor and copy assignment?
**Difficulty:** #advanced  
**Category:** #syntax #performance #interview_favorite  
**Concepts:** #move_constructor #copy_constructor #compiler_generated #rule_of_five

**Answer:**
The compiler does **not** generate the copy constructor or copy assignment if a move constructor or move assignment is explicitly defined.

```cpp
class A {
public:
    A(A&&) {}  // User-defined move constructor
    // Copy constructor NOT generated
    // Copy assignment NOT generated
};

A a1;
A a2 = a1;  // ❌ Error: no copy constructor
```

**Explanation:**
Defining move operations indicates custom resource handling that likely requires custom copy semantics too. The compiler won't generate potentially incorrect copy operations.

**Key takeaway:** Always define both copy and move operations when implementing either (Rule of 5).

#### Q3: If you define a destructor manually, what happens to move operations?
**Difficulty:** #advanced  
**Category:** #syntax #performance  
**Concepts:** #destructors #move_constructor #move_assignment #compiler_generated

**Answer:**
Move constructor and move assignment operator are **not** generated automatically when a destructor is user-declared.

```cpp
class A {
public:
    ~A() { /* cleanup */ }
    // Move constructor NOT generated
    // Move assignment NOT generated
};
```

**Explanation:**
The compiler assumes a custom destructor indicates resource management requiring custom move semantics. This prevents potentially incorrect default move operations.

**Key takeaway:** When defining destructors, explicitly define or default move operations per Rule of 5.

#### Q4: Is defining a destructor enough to suppress move semantics? Why or why not?
**Difficulty:** #advanced  
**Category:** #syntax #performance  
**Concepts:** #destructors #move_semantics #compiler_generated

**Answer:**
Yes. If a destructor is user-declared, move constructor and move assignment will not be generated by the compiler.

**Explanation:**
This design decision prevents the compiler from generating moves that might be incorrect when custom cleanup logic exists. You must explicitly define or default the move operations if needed.

**Key takeaway:** User-declared destructors suppress move generation; use `= default` to restore it if appropriate.

#### Q5: Why is it good practice to explicitly `=default` special member functions?
**Difficulty:** #intermediate  
**Category:** #design_pattern #syntax  
**Concepts:** #defaulted_functions #compiler_generated #best_practices

**Answer:**
Explicit `=default` improves code clarity, signals intent, helps with templates, and ensures compiler still generates them correctly.

```cpp
class A {
public:
    A() = default;
    A(const A&) = default;
    A& operator=(const A&) = default;
    A(A&&) = default;
    A& operator=(A&&) = default;
    ~A() = default;
};
```

**Explanation:**
- Makes intentions explicit and self-documenting
- Ensures correct generation in template contexts
- Prevents accidental suppression of needed operations
- Better for code maintenance and review

**Key takeaway:** Explicit is better than implicit for special member functions.

#### Q6: What happens if you delete only the move constructor? How does it affect return-by-value?
**Difficulty:** #advanced  
**Category:** #syntax #performance  
**Concepts:** #move_constructor #deleted_functions #copy_elision #return_value

**Answer:**
The class becomes move-incompatible. Return-by-value situations without copy elision lead to compilation errors.

```cpp
class A {
public:
    A(A&&) = delete;
};

A create() {
    return A();  // ❌ Error if elision doesn't apply
}
```

**Explanation:**
With deleted move constructor and no copy constructor, returning by value fails unless guaranteed copy elision (C++17+) applies. The move attempt is deleted, and there's no fallback.

**Key takeaway:** Deleting move without providing copy makes the type difficult to return by value in pre-C++17.

#### Q7: Does a class holding a `std::unique_ptr<T>` need a user-defined copy constructor? Why or why not?
**Difficulty:** #intermediate  
**Category:** #design_pattern #memory  
**Concepts:** #copy_constructor #unique_ptr #move_only #smart_pointers

**Answer:**
Yes, because `unique_ptr` is move-only, so copy operations are automatically deleted. If copying is needed, you must define custom deep-copy logic.

```cpp
class A {
    std::unique_ptr<int> ptr;
public:
    // Copy constructor deleted automatically
    // A(const A&) = delete;
    
    // If copying needed:
    A(const A& other) : ptr(std::make_unique<int>(*other.ptr)) {}
};
```

**Explanation:**
`unique_ptr` cannot be copied (exclusive ownership semantics). The compiler deletes copy operations by default. To allow copying, manually implement deep copy.

**Key takeaway:** Move-only members make the entire class move-only unless you implement custom copy logic.

#### Q8: If a base class has deleted copy constructor, what happens to the derived class copy constructor?
**Difficulty:** #intermediate  
**Category:** #syntax #inheritance  
**Concepts:** #copy_constructor #deleted_functions #inheritance

**Answer:**
The derived class's copy constructor is implicitly deleted too.

```cpp
class Base {
public:
    Base(const Base&) = delete;
};

class Derived : public Base {
    // Copy constructor implicitly deleted
};

Derived d1;
Derived d2 = d1;  // ❌ Error: copy constructor deleted
```

**Explanation:**
Derived class copy constructor must call base copy constructor. If base copy is deleted, derived copy cannot be generated.

**Key takeaway:** Deleted base copy operations propagate deletion to derived classes.

#### Q9: How do you make a class move-only?
**Difficulty:** #intermediate  
**Category:** #design_pattern #syntax  
**Concepts:** #move_only #deleted_functions #move_constructor #unique_ownership

**Answer:**
Delete copy operations and default or define move operations.

```cpp
class MoveOnly {
public:
    MoveOnly() = default;
    MoveOnly(const MoveOnly&) = delete;
    MoveOnly& operator=(const MoveOnly&) = delete;
    MoveOnly(MoveOnly&&) = default;
    MoveOnly& operator=(MoveOnly&&) = default;
};
```

**Explanation:**
Move-only types enforce unique ownership semantics, useful for resources like file handles, network sockets, or unique pointers.

**Key takeaway:** Move-only types prevent accidental copying of exclusive resources.

#### Q10: When do you prefer `=default` over writing out constructors? When is it dangerous?
**Difficulty:** #intermediate  
**Category:** #design_pattern #syntax  
**Concepts:** #defaulted_functions #best_practices #shallow_copy

**Answer:**
Use `=default` when the default behavior is correct. It's dangerous when managing raw pointers or resources requiring deep copy.

```cpp
class Good {
    std::vector<int> data;
public:
    Good(const Good&) = default;  // ✅ Safe: vector handles itself
};

class Bad {
    int* data;
public:
    Bad(const Bad&) = default;  // ❌ Dangerous: shallow copy!
};
```

**Explanation:**
`=default` is safe when all members are RAII types. With raw pointers, defaulted copy creates shallow copies leading to double deletion.

**Key takeaway:** Default works well for RAII classes; write custom logic for raw resource management.

#### Q11: Design a class that manages a dynamic resource. What special member functions will you write?
**Difficulty:** #intermediate  
**Category:** #design_pattern #memory #interview_favorite  
**Concepts:** #rule_of_five #resource_management #deep_copy #move_semantics

**Answer:**
Implement all five special members following Rule of 5.

```cpp
class Resource {
    int* ptr;
public:
    Resource(int val) : ptr(new int(val)) {}
    
    ~Resource() { delete ptr; }
    
    Resource(const Resource& r) : ptr(new int(*r.ptr)) {}
    
    Resource& operator=(const Resource& r) {
        if (this != &r) {
            delete ptr;
            ptr = new int(*r.ptr);
        }
        return *this;
    }
    
    Resource(Resource&& r) noexcept : ptr(r.ptr) {
        r.ptr = nullptr;
    }
    
    Resource& operator=(Resource&& r) noexcept {
        if (this != &r) {
            delete ptr;
            ptr = r.ptr;
            r.ptr = nullptr;
        }
        return *this;
    }
};
```

**Explanation:**
- Destructor frees resource
- Copy operations perform deep copy
- Move operations transfer ownership efficiently
- Self-assignment checks prevent bugs

**Key takeaway:** Resource-managing classes need complete Rule of 5 implementation.

#### Q12: How would you implement deep copy semantics in a class that holds a raw pointer?
**Difficulty:** #intermediate  
**Category:** #memory #design_pattern  
**Concepts:** #deep_copy #copy_constructor #assignment_operator #pointers

**Answer:**
Allocate new memory and copy the pointed-to data, not the pointer itself.

```cpp
class DeepCopy {
    int* data;
public:
    DeepCopy(const DeepCopy& other) {
        data = new int(*other.data);  // Allocate and copy
    }
    
    DeepCopy& operator=(const DeepCopy& other) {
        if (this != &other) {
            delete data;
            data = new int(*other.data);
        }
        return *this;
    }
};
```

**Explanation:**
Shallow copy copies pointer values, causing both objects to point to the same memory. Deep copy allocates independent memory for each object.

**Key takeaway:** Always use deep copy for pointer members to avoid double deletion and data corruption.

#### Q13: You have a class with move-only semantics. How would you use it in a standard container?
**Difficulty:** #intermediate  
**Category:** #design_pattern #performance  
**Concepts:** #move_only #containers #move_semantics

**Answer:**
Standard containers like `std::vector` can handle move-only types as long as they have valid move operations.

```cpp
class MoveOnly {
public:
    MoveOnly() = default;
    MoveOnly(const MoveOnly&) = delete;
    MoveOnly(MoveOnly&&) = default;
};

std::vector<MoveOnly> vec;
vec.push_back(MoveOnly());  // ✅ Uses move
vec.emplace_back();  // ✅ Constructs in-place
```

**Explanation:**
Containers use move operations when adding elements. Move-only types work fine as long as they're not copied.

**Key takeaway:** Move-only types integrate seamlessly with standard containers.

#### Q15: You inherit from a base class with a non-virtual destructor. Is this safe? When might it break Rule of 5?
**Difficulty:** #advanced  
**Category:** #memory #inheritance #interview_favorite  
**Concepts:** #destructors #virtual_functions #inheritance #undefined_behavior

**Answer:**
Not safe if you delete derived objects through base pointers. It causes undefined behavior and resource leaks.

```cpp
class Base {
    ~Base() {}  // ❌ Non-virtual
};

class Derived : public Base {
    int* data;
    ~Derived() { delete data; }
};

Base* ptr = new Derived();
delete ptr;  // ❌ UB: Only Base destructor called, data leaked
```

**Explanation:**
Without virtual destructor, only the base portion is destroyed. Derived resources are never cleaned up.

**Key takeaway:** Always use virtual destructors in polymorphic base classes.

#### Q16: How would you make a class both polymorphic and Rule-of-5-safe?
**Difficulty:** #advanced  
**Category:** #design_pattern #memory  
**Concepts:** #polymorphism #rule_of_five #virtual_functions #destructors

**Answer:**
Use virtual destructor and implement all five special members with proper deep copy/move logic.

```cpp
class PolymorphicResource {
protected:
    int* data;
public:
    virtual ~PolymorphicResource() { delete data; }
    
    PolymorphicResource(const PolymorphicResource& other)
        : data(new int(*other.data)) {}
    
    PolymorphicResource& operator=(const PolymorphicResource& other) {
        if (this != &other) {
            delete data;
            data = new int(*other.data);
        }
        return *this;
    }
    
    PolymorphicResource(PolymorphicResource&& other) noexcept
        : data(other.data) {
        other.data = nullptr;
    }
    
    PolymorphicResource& operator=(PolymorphicResource&& other) noexcept {
        if (this != &other) {
            delete data;
            data = other.data;
            other.data = nullptr;
        }
        return *this;
    }
    
    virtual void process() = 0;
};
```

**Key takeaway:** Combine virtual destructor with complete Rule of 5 for safe polymorphic resource management.

#### Q17: If you define both copy and move constructors, how does the compiler decide which to use?
**Difficulty:** #intermediate  
**Category:** #syntax #performance  
**Concepts:** #move_constructor #copy_constructor #overload_resolution #rvalue

**Answer:**
If the expression is an rvalue (temporary or moved), move is preferred. If lvalue, copy is used.

```cpp
class A {
public:
    A(const A&) { std::cout << "Copy\n"; }
    A(A&&) { std::cout << "Move\n"; }
};

A a1;
A a2 = a1;           // Copy (lvalue)
A a3 = std::move(a1); // Move (rvalue)
A a4 = A();          // Move (temporary)
```

**Explanation:**
Overload resolution prefers exact matches. Rvalues bind to rvalue references (`T&&`), triggering move. Lvalues bind to lvalue references (`const T&`), triggering copy.

**Key takeaway:** Move is an optimization for temporaries; copy is used for persistent objects.

#### Q18: Why should you implement both move and copy assignment together if you implement one?
**Difficulty:** #intermediate  
**Category:** #design_pattern #syntax  
**Concepts:** #rule_of_five #copy_assignment #move_assignment

**Answer:**
If you implement one, the other is not automatically generated. Rule of 5 mandates explicit intent for all operations.

```cpp
class A {
public:
    A& operator=(const A&) { return *this; }
    // Move assignment NOT generated
};

A a1, a2;
a1 = std::move(a2);  // Uses copy assignment (less efficient)
```

**Explanation:**
Without move assignment, `std::move` falls back to copy assignment, losing performance benefits. Complete implementation ensures correct semantics for all cases.

**Key takeaway:** Implement all operations explicitly to avoid performance degradation.

#### Q19: How does Rule of 5 apply to a Pimpl (Pointer-to-Implementation) class?
**Difficulty:** #advanced  
**Category:** #design_pattern #performance  
**Concepts:** #pimpl #rule_of_five #forward_declaration #incomplete_types

**Answer:**
Pimpl requires explicit definitions of all five special members in the implementation file where the impl type is complete.

```cpp
// Header
class Widget {
    struct Impl;
    std::unique_ptr<Impl> pImpl;
public:
    Widget();
    ~Widget();
    Widget(const Widget&);
    Widget& operator=(const Widget&);
    Widget(Widget&&) noexcept;
    Widget& operator=(Widget&&) noexcept;
};

// Implementation
Widget::~Widget() = default;
Widget::Widget(Widget&&) noexcept = default;
Widget& Widget::operator=(Widget&&) noexcept = default;

Widget::Widget(const Widget& other)
    : pImpl(std::make_unique<Impl>(*other.pImpl)) {}

Widget& Widget::operator=(const Widget& other) {
    *pImpl = *other.pImpl;
    return *this;
}
```

**Explanation:**
The impl type is incomplete in the header. Default implementations requiring deletion/copying must be in the implementation file where the type is complete.

**Key takeaway:** Pimpl requires out-of-line definitions of special members in the implementation file.

#### Q20: Is it safe to rely solely on defaulted move operations? When might this lead to bugs?
**Difficulty:** #advanced  
**Category:** #memory #design_pattern  
**Concepts:** #move_semantics #defaulted_functions #shallow_move #resource_management

**Answer:**
Not safe if the class has complex invariants or pointer aliasing. Defaulted moves perform shallow moves which may leave objects in invalid states.

```cpp
class HasInvariant {
    int* begin;
    int* end;
public:
    // Defaulted move breaks invariant if begin/end must stay synchronized
    HasInvariant(HasInvariant&&) = default;
};
```

**Use cases requiring custom moves:**
- Classes with internal pointers to own members
- Intrusive data structures
- Objects with complex invariants between members
- Classes using custom allocators

**Key takeaway:** Always validate that defaulted moves maintain class invariants; implement custom moves when needed.
